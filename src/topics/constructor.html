Constructor / Prototype Pattern

The problem

The constructor pattern is useful when we need to create objects that share the same methods and property names.

By taking advantage of javascript prototypes, we save memory by using the same reference for methods in all our objects.

Constructors

Custom

      
        function ConstructAStudent(name, age) {
          this.name = name;
          this.age = age;
        }

        const student1 = new ConstructAStudent('Juan', 20);
        const student2 = new ConstructAStudent('Susana', 25);

Any function can be a constructor when called with the new keyword. The this will be the context for the new object, so any properties or methods defined to it will be part of the returned object.

Constructors

Prototypes

      
        const proto = {
          greet() {
            return 'hello!'
          }
        };

        const Person = function(name) {
          this.name = name;
        }

        Person.prototype = proto;

        const student = new Person('Juan');

        student.greet();
        // 'hello!'

All functions have a prototype property that does not return the function's property, but defines the prototype for any object created when that function is used as a constructor.

Constructors

Classes

      
        class Student {
          constructor(name, age) {
            this.name = name;
            this.age = age;
          }
          sayHi() {
            return `Hi! My name is ${this.name}.`;
          }
        }

        const student1 = new Student('Juan', 20);

        student1.sayHi();
        // Hi! My name is Juan.

A class is defined using the class keyword, a name, and curly braces, everything inside the code block will run in strict mode, even if it is not set in the code.
An instance of the class can be created using the new keyword.
